### 1.2 执行规则

所有角色遵循以下规则：

1. **每个角色文件包含 `执行工作流` 章节** ─ 按编号步骤执行，不可跳步
2. **每步有明确产出** ─ 产出物路径、格式、通过标准
3. **激活 Skill 有明确时机** ─ 不在进入阶段时一股脑加载所有 skill，而是在需要时才激活
4. **交接前完成自检** ─ 使用 `self-review` skill 逐项检查
5. **通信使用统一协议** ─ `communication-protocol` 中定义的格式

### 1.3 并行规则

| 并行组 | 角色 | 前置条件 |
|--------|------|---------|
| 制作并行组 | Interaction / UI / UX / VA / Animator / Sound | Designer 完成数据结构和公式后 |
| 交付并行组 | Developer / Test Architect | UI/UX 冻结签字后 |
| 审计链 | Developer → SecAuditor → QA Guardian | 代码审查通过后串行 |
| 全域持续 | Meta-Guardian | 任何阶段均可激活 |

## 二、8 阶段流水线

**标准通道：**
```
0.构思 → 1.规划 → 2.设计 → 3.交互+视觉 → 4.开发TDD → 4a.安全审计 → 5.质量审计 → 6.发布 → 7.复盘
```

| 阶段 | 负责 | 质量分 | 主要活动 |
|------|------|--------|---------|
| 0 | Producer | — | brainstorming 收敛需求 |
| 1 | Producer+Architect+Director | ≥70 | PRD + ADR + 启动会 |
| 2 | Designer | ≥75 | GDD + 数值验证 |
| 3 | Interaction+UI+UX+VA+Animator | ≥80 | 交互+UI冻结 |
| 4 | Developer | ≥80 | TDD循环 + 代码审查 |
| 4a | Security Auditor | ≥80 | 安全审计（代码审查后） |
| 5 | QA Guardian | ≥85 | 功能+边界+回归测试 |
| 6 | DevOps | ≥90 | 版本标签 + 部署验证 |
| 7 | Meta-Guardian+全员 | — | 复盘 + 改进计划 |

**快速通道:** `Bug修复 → 4.开发 → 4a.安全审计 → 5.质量审计 → 6.发布` (详见 `pipeline/fast-track.md`)

---

## 三、质量门

4 维评分（完整性/正确性/规范性/可交接性），加权平均。低于分数线自动驳回。

评分工: 执行方自评 + 审查方独立评分 → 取平均。

---

## 四、交接协议（6 项）

1. 输入物存在且版本正确
2. 输出物完整
3. 质量评分 ≥ 最低分数线
4. 未决项已记录
5. 下阶段角色已确认接收
6. 上游角色保留消息通道（2h 内可提问）

---

## 五、通信协议

使用统一格式 `@from/@to/@priority/@subject/@body/@response-needed/@deadline`。
通道: 阶段内→直接、跨阶段→Producer中转、P0→直接+Producer同步。

---

## 六、快速通道

Bug修复/UI微调/数值调整/P0热修复 → Producer一次审批，跳过阶段 0-3。
每月限 5 次，连续 3 次触发 Meta 审计。

---

## 七、AI 编程规范

所有代码遵循 `AI编程规范.md`:
- 文件头 `@file/@role/@handoff` 标签
- 函数注释 `@param/@returns/@throws`
- 命名: 变量 camelCase / 常量 UPPER_SNAKE / 文件 kebab-case
- 模块: init/destroy 接口 + EventBus 通信
- 禁止: var/console.log(生产)/硬编码中文/无注释 Magic Number

---

## 八、审计追踪

Meta-Guardian 持续审计:
- 每阶段完成时 ─ 交付物完整性检查
- 每次交接时 ─ 6 项交接协议执行检查
- 每次发布时 ─ 完整质量审计
- 每季度 ─ 全面技能盘点
