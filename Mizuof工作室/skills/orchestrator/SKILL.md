## 1. 质量评分引擎

### 评分流程

```
执行方自评 ──→ 审查方独立评分 ──→ 取平均 ──→ 判断
   ↓                ↓               ↓          ↓
 0-100分         0-100分         均分      ≥分数线? 推进 : 驳回
```

### 评分表

每次评分填写此表：

```markdown
## 阶段 {N} 质量评分

| 维度 | 权重 | 执行方自评 | 审查方评分 | 均分 |
|------|------|-----------|-----------|------|
| 完整性 | 30% | /100 | /100 | |
| 正确性 | 30% | /100 | /100 | |
| 规范性 | 20% | /100 | /100 | |
| 可交接性 | 20% | /100 | /100 | |
| **加权总分** | | | | **/100** |

分数线: {分数线}
结果: 通过 / 驳回
```

### 阶段分数线

| 阶段 | 分数线 | 执行方 | 审查方 |
|------|--------|--------|--------|
| 1 规划 | 70 | Producer | Architect |
| 2 设计 | 75 | Designer | Producer |
| 3 交互+视觉 | 80 | UI/UX | Developer |
| 4 开发TDD | 80 | Developer | Test Architect |
| 5 质量审计 | 85 | QA Guardian | Producer |
| 6 发布 | 90 | DevOps | QA Guardian |

## 3. 快速通道审批

### 触发条件检查

- [ ] Bug 修复（改动文件数 ≤ 3）
- [ ] UI 微调（仅 CSS 变更）
- [ ] 数值调整（仅数据表变更）
- [ ] P0 热修复（Producer 判定）

### 审批流程

1. Producer 审批 → 批准/拒绝
2. 通过后跳过阶段 0-3
3. 记录快速通道使用次数
4. 连续 3 次 → 触发 Meta-Guardian 审计

---

## 4. 驳回恢复

| 驳回原因 | 恢复操作 | 时限 |
|---------|---------|------|
| 完整性不足 | 列出缺失项 → 返回角色补齐 | 4h |
| 正确性不达标 | 列出失败测试 → 返回修复 → 重跑失败项 | 4h |
| 规范性不达标 | 列出违规项 → 返回修正 | 2h |
| 可交接性不足 | 下游列出不清之处 → 上游补充文档 | 2h |
| 第 3 次驳回 | Producer 介入 → 拆解或重分配 | 立即 |
